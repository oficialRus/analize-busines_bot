# Geo Analysis Dashboard

MVP панели геоаналитики на **Next.js 14+**, **TypeScript**, **Tailwind CSS**, **shadcn/ui**, **react-hook-form**, **Zod**. Сеточный анализ конкурентов через **2GIS Places API**, выбор города и bbox через **Nominatim + Overpass**. История успешных прогонов хранится в **SQLite** на сервере разработки.

## Требования

- Node.js 18+
- npm

Сборка **better-sqlite3** требует нативных инструментов (на Windows обычно достаточно установленного Node + MSVC Build Tools при ошибках компиляции).

## Запуск

```bash
npm install
npm run dev
```

Откройте [http://localhost:3000](http://localhost:3000).

Сборка и прод:

```bash
npm run build
npm start
```

## Переменные окружения

```bash
cp .env.example .env.local
```

- **`DGIS_API_KEY`** — ключ 2GIS (только сервер, в клиент не попадает).
- **`ANALYSIS_DB_PATH`** (опционально) — путь к файлу SQLite; по умолчанию `data/analysis-history.db`.

## Как устроен сеточный анализ

1. По bbox города строится регулярная сетка (`lib/analysis/grid.ts`).
2. Для **центра каждой ячейки** выполняется запрос к 2GIS (последовательно, чтобы не перегружать API).
3. **Дедупликация** (`lib/analysis/dedupe.ts`): один и тот же объект может попасть в несколько соседних кругов поиска. При обходе ячеек в фиксированном порядке учитывается только **первое вхождение** id конкурента. Тогда:
   - метрики ячейки (число конкурентов, средний рейтинг, сумма отзывов, топ имён) считаются по **уникальным в рамках этой ячейки после глобального дедупа**;
   - **`totalCompetitorsFound`** = число уникальных id на всей сетке;
   - сумма `competitorsCount` по всем ячейкам совпадает с `totalCompetitorsFound`.
4. **Score** (`lib/analysis/scoring.ts`) — индекс 0–100 «привлекательности для точки входа»: спрос проксируется логарифмом суммы отзывов, насыщение — числом конкурентов, корректировка — средним рейтингом. Точная формула и коэффициенты описаны в комментариях в `scoring.ts`.
5. **Рекомендации** — отдельный эвристический слой по порогам (высокая конкуренция + отзывы, умеренная зона для теста, низкая активность, плотный рынок и т.д.).
6. **Топ зоны** — ячейки с наибольшим score; для каждой зоны возвращаются `shortRecommendation` и развёрнутый `summaryText`.

## Ограничения demo / 2GIS

- Лимиты выдачи и квоты API зависят от тарифа; в коде на ячейку запрашивается до **50** объектов.
- Качество и полнота данных зависят от региона и типа запроса.
- Последовательный обход большой сетки увеличивает время ответа — для v2 имеет смысл батчинг, кэш и фоновые задачи.

## История анализов (SQLite)

- После **успешного** `POST /api/analysis/run` результат сохраняется в БД (сбой записи не отменяет ответ клиенту, ошибка пишется в лог сервера).
- `GET /api/analysis/history?limit=30` — список метаданных.
- `GET /api/analysis/[id]` — полный JSON результата (`requestId` = id записи).

В **serverless** (например, Vercel) файловая SQLite обычно **не сохраняется** между инвокациями — для продакшена заложена замена хранилища (Postgres / S3 + id), интерфейс слоя истории можно сохранить.

## Что улучшить в v2

- Параллельные запросы с ограничением concurrency, ретраи и кэш по ячейкам.
- Карта тепла по ячейкам, клик по ячейке → детализация.
- Учёт категорий 2GIS, фильтры по рейтингу/дистанции.
- Персистентная БД в облаке, мультипользовательский доступ.
- Калибровка score под верифицированные бизнес-метрики (не только эвристики).

## Структура (кратко)

| Путь | Назначение |
|------|------------|
| `app/api/analysis/run` | Запуск сеточного анализа |
| `app/api/analysis/history` | Список истории |
| `app/api/analysis/[id]` | Восстановление результата |
| `lib/analysis/*` | Сетка, bbox, дедуп, скоринг, история |
| `lib/providers/2gis/*` | Клиент 2GIS |
| `lib/geo/*` | Nominatim, Overpass, bbox города |
| `features/analysis/*` | Форма, результаты, история в UI |
| `features/geo/*` | Выбор провайдера, город/районы |
| `types/` | Доменные типы |
